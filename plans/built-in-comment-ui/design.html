Design: built-in-comment-ui

Scenario: built-in-comment-ui · Plugin: plan-harness · collaboration review self-contained

1. Executive Summary

Today plan-harness generates high-quality HTML plan docs and can share them over a devtunnel with a password gate, but the reading experience is strictly read-only. Reviewers who spot a problem have no way to write back — they fall out of the tool into Slack / Teams / email / a call. The author has no durable record of which sentence in which section prompted the pushback. For a planning tool whose whole value proposition is reviewable before you code, the missing write path is a first-class gap.

This scenario adds a text-selection commenting layer native to every rendered plan doc. Reviewers select a span, click a floating Comment control, and leave a threaded comment attributed to the display name they signed in with. Comments persist as append-only JSONL on the host filesystem using a W3C-style anchor (prefix + exact + suffix) so they survive regeneration. No database, no SaaS, no OAuth — the same auth.js password gate guards writes.

Stakeholders

  • Host / Author — runs plan-harness locally, gets a durable, attributed backlog tied to exact sentences.
  • Reviewer — anyone with URL + password, logs in with display name only, no account.
  • Agent personas (future-ready) — architect / PM / tester agents that may respond in /plan-review cycles; schema accommodates agent authorship even if the MVP writes are human-only.
  • plan-harness contributor — developer in this repo reading the plan docs about this plugin; self-development recursion is a real UX constraint.

Success Metrics

  • TTFC (time-to-first-comment) p50 < 30s / p95 < 90s from landing on login.
  • Anchor survival ≥ 90% after doc regen.
  • Author sees per-scenario unresolved count at a glance on dashboard.
  • Zero silent comment loss under disk-full / concurrent / tunnel-drop.
  • Keyboard parity — every create / reply / resolve / delete reachable via keyboard with ARIA-announced result.

2. Core Design Principles

Open decision Who can resolve a thread?

The Architect proposes that any authenticated user may resolve or unresolve any thread (matches Google Docs / Notion model; enables reviewer-driven cleanup). The PM proposes host-only resolve (AC-4 P0; prevents reviewers marking-as-resolved without author acknowledgement).

Status: Unresolved — flagged for user decision before implementation begins. Impacts the permission matrix in §9 and AC-4 in §7.

Open decision Fuzzy re-anchor algorithm

The Architect specifies Jaccard token-overlap ≥ 0.72 (cheap, whitespace-tokenized, good for whole-sentence survival). The PM specifies Levenshtein similarity ≥ 0.85 (character-level, catches small edits but costlier on long selections).

Status: Unresolved — pick before coding the re-anchor cascade. Affects tier (c) in §4 Re-anchor sequence and UC-6 in §7.

  1. HTML is the source of truth for anchors. Every <h2>/<h3>/<h4> in every generated plan doc gets a stable, deterministic id="sec-<16hex>" derived from a canonical hash of its heading path. Comments anchor into those sections with prefix + exact + suffix. No separate anchor store, no fragile XPath.
  2. Server-side identity is non-negotiable. author is stamped from req.user.name (or the sentinel "Host (local)" on loopback). Any client-supplied author is silently dropped. The session cookie already exists; no new auth primitive is introduced.
  3. Re-anchor resilience over anchor purity. When a doc regenerates, a three-tier match runs: exact prefix+exact+suffix in the same sectionId → exact text anywhere in the same section (records migratedFrom) → token-overlap Jaccard ≥ 0.72 across sections. Anything below that threshold is flagged orphaned. Orphans > wrong anchors, always.
  4. Append-only JSONL; never a destructive write. Edits, resolves, deletes, and re-anchors all append new events to plans/<scenario>/.comments/<doc-slug>.jsonl. Last event per id wins on read (event-sourced). Deletes are tombstones; bodies are never rewritten on disk.
  5. Zero new runtime dependencies. Node stdlib only beyond the existing @modelcontextprotocol/sdk. SSE over node:http (not WebSocket — would need an upgrade dance or third-party lib). Crypto via node:crypto. The widget is inlined into every generated HTML doc by base.js, matching the existing self-contained-HTML invariant; no separate script endpoint, no CDN, offline viewing still works.
  6. Safety by default. Path params regex-validated (^[a-zA-Z0-9_-]+$, ≤ 80 chars). Resolved paths must start with workspaceRootPath + sep + 'plans' + sep. Bodies stored raw, HTML-escaped on render. Session cookie is SameSite=Strict; HttpOnly; Secure — relied upon for CSRF with no added token (the marginal security gain does not justify the staleness failure mode).
  7. Never silent-fail. Disk-full yields a 507 with a user-visible message; the client surfaces a draft-saved-locally banner and auto-retries with exponential backoff. A 200 without a durable write is prohibited.

4. Architecture

Three cooperating parts:

  1. Browser widget (~350 LOC inline) baked into every generated HTML doc by base.js. Handles selection detection, anchor serialization, sidebar rendering, and API calls.
  2. comment-manager.js — new server module: JSONL read / append / patch, re-anchor logic, SSE client registry, token-bucket rate limiter.
  3. 5 new HTTP routes + 1 SSE route on web-server.js, gating all mutations behind the existing verifyCookie / loopback-bypass auth gate.

Technology Choices

SSE vs polling vs WebSocket decision: SSE

node:http supports SSE natively (chunked text/event-stream); WebSocket needs an upgrade dance or a third-party lib — rejected because it violates the no-new-deps rule. 5-second polling lags multi-reviewer sessions. SSE gives sub-second delivery with a single persistent connection per open doc. An in-memory Map<docKey, Set<Response>> registry flushes on writes; 30s pings keep the tunnel alive; EventSource auto-reconnects on the client.

Resolve semantics decision: collapse + flag

Resolved threads fade in the sidebar with a Resolved badge; click to expand; the thread is never removed. Matches Google Docs / Word / Notion. Who may resolve is the Open Decision flagged above.

Error Format

All errors: { "error": "CODE", "message": "human readable" }.

CodeHTTPMeaning

Rate Limiting & Path Safety

  • Rate limit: 5 POST per session per 10s via in-memory token bucket (Map<sid, { tokens, lastRefill }>). 429 with Retry-After on empty.
  • Path safety: route params match ^[a-zA-Z0-9_-]+$, max 80 chars. Resolved filesystem path must start with workspaceRootPath + sep + 'plans' + sep, mirroring the existing /view fix.

6. UX Design

Primary User Flow

  1. Reviewer lands on a plan doc (via tunnel URL or localhost).
  2. Reviewer drags to select ≥ 3 chars inside a section that has a [data-section-id] ancestor.
  3. A floating Comment button appears within 150ms, anchored to the selection end (Word / Notion / Hypothesis model). Keyboard: Ctrl+Alt+M. No right-click menu in MVP.
  4. Click (or keyboard shortcut) opens a right-side sliding panel (~380 px, non-modal). The selection highlights; composer textarea takes focus.
  5. Ctrl+Enter submits. Success appends to JSONL, renders an inline <mark> highlight + sidebar thread card. Failure shows an inline error and re-enables submit after 2s (5xx) or on a visible countdown (429).
  6. Esc closes the composer; focus returns to the selection end. Drafts are auto-saved to localStorage keyed by (scenario, doc, anchor) and restored on reload.

Why a side panel over a modal or inline bubble? Threads get long; modals trap focus and block re-reading the doc; inline bubbles clip at section boundaries and fight embedded charts. A non-modal slide-over lets the reviewer re-check context while writing. Below 720 px, the panel becomes a bottom sheet.

7. Use Cases & Acceptance Criteria

Use Cases

UC-1 — Reviewer leaves a comment on a text selection MVP

Actor: Reviewer. Trigger: selects ≥ 3 chars inside a [data-section-id]. Pre: logged in, doc loaded.

  1. Drag-select span.
  2. Floating Comment button appears within 150ms anchored to selection end.
  3. Click button or Ctrl+Alt+M.
  4. Side panel opens; composer focused; selection highlighted.
  5. Type body + Ctrl+Enter.
  6. Client POSTs; server validates auth, appends JSONL, 201s.
  7. UI renders highlight + thread card; composer clears.

Alt: 4a. Esc → panel closes, no write. Post: one JSONL line; visible on next poll / SSE event.

UC-2 — Reviewer replies to a thread MVP

Actor: Reviewer. Click thread in sidebar or inline highlight → thread expands → type reply → POST with replyTo: <parent-id> → server appends → UI nests the reply. Post: thread +1 msg; reply inherits threadId.

UC-3 — Host reads new comments MVP

Scenario cards show N new / M unresolved badges → click scenario → per-doc breakdown → open doc with sidebar expanded, newest unread pinned → viewing a thread advances the host's localStorage read cursor. Post: cursor updates; badges refresh next load.

UC-4 — Host resolves a thread Open decision

Actor: Host (loopback OR role: host session). Click Resolve → 5s undo toast → on timeout, POST resolve → thread collapses with Resolved badge. Alt: click Undo → no write. Post: { resolved: true } event appended; thread still in JSONL.

Note: PM requires host-only; Architect permits any auth user. See Open Decision in §2.

UC-5 — Reviewer edits / deletes own comment MVP

Actor: Reviewer (author match + within 10-min edit window). Edit: inline editor → PATCH → append-only edit event (prior body preserved). Delete: tombstone event → UI shows "[deleted by author]". Post: no destructive disk write.

UC-6 — Regen + re-anchor MVP

Actor: Host. Doc regenerated with preserved sectionIds → on next load, re-anchor: exact prefix+exact+suffix in same sectionId → hold; else fuzzy (Levenshtein 0.85 PM / Jaccard 0.72 Architect) within section → hold; else mark orphaned. Post: each comment has status anchored / fuzzy / orphaned.

UC-7 — Orphan recovery MVP

Pinned "Needs reattachment (N)" at top of sidebar → click orphan shows original exact in a blockquote → Reattach to selection mode → select new text → reanchor event. Alt: Archive hides without a new anchor. Post: orphan cleared.

UC-8 — Badge on scenario card MVP

Dashboard shows unresolved counts (document state, not per-user). Scenario detail has per-doc breakdown. Clicking a doc with unresolved scrolls to first unresolved highlight. Post: reviewer's cursor updates on thread view.

UC-9 — Export PDF with margin notes P2

Click Export PDF on doc → pipeline inlines threads as numbered footnotes referencing highlighted spans; toggle to export plan-only. Default includes comments; resolved are dimmed. Post: static PDF with review trail.

UC-10 — Keyboard-only a11y P1

Actor: Screen-reader reviewer. Tab to section → Enter reading mode → arrow keys advance selection by word/sentence → Ctrl+Alt+M opens composer, focus in textarea, SR announces "Comment composer, referencing: <selected text>" → Ctrl+Enter submits → ARIA live region announces "Comment posted". Alt: Esc closes composer; focus returns to anchor. Post: no mouse used; focus never lost.

UC-11 — Empty state MVP

First visit, zero comments → sidebar shows empty-state "No comments yet. Select any text to leave the first." plus a one-time dismissible tooltip on first selection. Post: reviewer discovers primary action without docs.

UC-12 — Tunnel dropout during draft P1

Offline detected or POST fails → draft saved to localStorage keyed by (scenario, doc, anchor) → banner "Saved as draft — will retry when reconnected" → auto-retry w/ exponential backoff (3 tries, 1s / 2s / 4s). Reload restores draft + re-highlights selection. Post: zero draft loss.

CSRF Posture

Decision No HMAC CSRF token

The session cookie is already SameSite=Strict; HttpOnly; Secure. Cross-site form / fetch with cookies is blocked by the browser. Adding a CSRF token buys minimal marginal security in this threat model while costing 40+ lines of server code and introducing a cache-staleness failure mode. If the threat model changes (e.g., SameSite relaxed), revisit.

Attribution Integrity

author is stamped server-side from req.user (or the loopback sentinel "Host (local)"). Any client-supplied author field is ignored silently. This is enforced in one place in comment-manager.appendComment, not by route-level validation, to eliminate the class of "forgot to validate on the new route" bugs.

Path Safety

  • Route params must match ^[a-zA-Z0-9_-]+$ and be ≤ 80 chars.
  • Resolved filesystem path must startsWith(plansRoot + sep) after path.resolve(). Rejects .., backslashes, null bytes, absolute paths.
  • .comments/ created via fs.mkdir({ recursive: true }).
  • Windows: the regex rejects both / and \; the startsWith check uses path.sep.

Audit Log

Every mutation writes a single line to stderr (unparsed by stdout JSON-RPC). Host can tee to a file if needed.

  • [comments] POST <scenario>/<doc> author=<name> id=<id>
  • [comments] PATCH <id> field=resolved value=true actor=<name>
  • [comments] DELETE <id> actor=<name>
  • [comments] reanchor <scenario>/<doc> held=N migrated=M orphaned=K
  • [comments] SSE client connected / disconnected (sid=...)
  • [comments] BLOCKED path-traversal param=<raw> ip=<...> (on rejection)

10. Risk Analysis

Likelihood (L) and Impact (I) are rated H / M / L.

R-1 Regen invalidates anchors → orphan explosion L:HI:H
A naive regen resets every id and every anchor becomes orphaned. Review trust collapses.
Mitigate: stable deterministic sectionIds; Levenshtein 0.85 / Jaccard 0.72 fuzzy tier; orphan bucket recoverable, not catastrophic; stderr telemetry; target orphan rate < 10%.
R-2 Reviewer spam L:MI:M
A bad actor (or a bored one) fires hundreds of comments over the tunnel.
Mitigate: rate limit 10/min + 100/hour per session; visible countdown; host plan_share_stop kicks everyone; future per-session purge.
R-3 Path traversal L:MI:H
../ or \\ in scenario / doc params could read or write outside plans/.
Mitigate: allowlist via plan_list_scenarios; reject .. / \\ / null / absolute; reuse scenario-name sanitizer; unit tests with a published attack list.
R-4 Loss on disk-full / concurrent writes L:MI:H
Simultaneous appendFile calls interleave, or ENOSPC silently drops data.
Mitigate: O_APPEND; per-file mutex; fsync; 507 on ENOSPC (never silent 200); 50-parallel stress test.
R-5 XSS via comment body L:MI:H
Reviewer submits <img onerror>; renders as script if stored and re-inserted raw.
Mitigate: escape on output via textContent; CSP default-src 'self'; payload-list unit test; same escape applied to author.
R-6 Reviewers don't discover the button L:MI:M
Selection → button is a learned pattern. First-timers may never find it.
Mitigate: first-visit tooltip; empty-state CTA in sidebar; ? cheat-sheet overlay; TTFC metric < 30s p50; iterate if it misses.
R-7 Tunnel drop loses drafts L:HI:M
devtunnel flakes mid-compose; the draft is gone.
Mitigate: localStorage draft keyed by (scenario, doc, anchor); auto-retry exp backoff (1s / 2s / 4s); banner + restore-on-reload.
R-8 Self-dev recursion L:LI:M
Editing plan-harness while reading its own design comments; commenting on a doc about the comment widget; reviewer noise leaking into the plugin repo.
Mitigate: document in README; separate port for self-dev vs consumer; .gitignore .comments/ in the plugin repo.
R-9 Password-shared tunnel exposes comments L:MI:M
Anyone with the tunnel URL + password can read (and write) every comment.
Mitigate: document clearly in plan_share output; per-share password rotation; session TTL enforced in auth.js; future revoke-session API.
R-10 JSONL grows unbounded long-term L:LI:L
10k comments ~ 2 MB; 100k ~ 20 MB. Eventually parsing gets slow.
Mitigate: acceptable at expected scale; compaction deferred; note in ROADMAP.md; revisit if we see > 10 MB in the wild.
