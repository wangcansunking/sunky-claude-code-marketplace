## Product Vision

`plan-harness` turns a rough idea into a reviewable, implementation-ready plan — as a set of interactive HTML documents generated by a team of agent personas.

It is **a generator and a local viewer**, not a collaboration platform.
Collaboration happens wherever the team already works (HTML dashboard via tunnel, or exports into Google Docs / Word / ADO Wiki / Notion / etc.).

Target users: any developer using Claude Code, not tied to any company or stack.

---

## Product Pillars

1. **High-quality generation** — designs, test plans, state machines, implementation plans, all as polished self-contained HTML
2. **Composable context** — scenarios inherit project contexts (three-tier loading), so the same codebase yields consistent plans
3. **Agent team review** — architect / PM / tester / writer personas critique each document, with iterative refinement
4. **Real-world integration** — publish to where reviewers are, not just where the plan was generated

## Collaboration Architecture (decided)

### Core (built-in, platform-agnostic)

- **HTML dashboard** remains source of truth (no markdown migration)
- **Built-in comment UI** — text-selection anchored, Word-like experience
- **Local filesystem storage** — `plans/<scenario>/.comments/<doc>.jsonl` (append-only)
- **Share via devtunnel** — existing `plan_share` with password protection
- **Universal exports** — PDF, `.docx`, single-file HTML

### Adapters (opt-in, pluggable)

Core ships with a small set; community can add more:

| Adapter | Reach | Priority |
|---|---|---|
| GitHub Gist / Discussions / Issues | Developer-universal | High |
| Google Docs | Cross-industry default | High |
| Notion | Startups / modern teams | High |
| Confluence | Enterprise | Medium |
| Linear | SaaS | Medium |
| Teams / Slack / Discord (notify only) | Async ping | Medium |
| Word / SharePoint / ADO Wiki | MSFT stacks | Low (one of many) |

Each adapter implements:
```js
export default {
  name: "google-docs",
  requiredCreds: ["GOOGLE_OAUTH_TOKEN"],
  push: async ({ scenario, docType, content }) => {...},
  pull: async ({ scenario, docType }) => {/* comments */},
};
```

The plugin does **not** manage OAuth tokens. Users set env vars themselves.

## Text-Selection Comment Design (sketch — needs review before impl)

The user selects any span of text in a rendered plan document, clicks a "Comment" popover button, and leaves a comment — the same mental model as Word / Google Docs / Notion / Hypothesis.

### Storage format

`plans/<scenario>/.comments/<doc-slug>.jsonl` — one JSON object per line, append-only:

```json
{
  "id": "cmt_7f3a9c",
  "createdAt": "2026-04-17T10:22:00Z",
  "author": "alice",
  "anchor": {
    "sectionId": "sec-api-contract",
    "exact": "the retry policy uses exponential backoff",
    "prefix": "For failed requests, ",
    "suffix": ". Max attempts is 5."
  },
  "body": "Shouldn't this be jittered to avoid thundering herd?",
  "threadId": "cmt_7f3a9c",
  "replyTo": null,
  "resolved": false
}
```

Anchor model is inspired by W3C TextQuoteSelector: `exact` + `prefix` + `suffix` gives robust re-anchoring even if the document is regenerated or edited.

### UI flow

1. User selects text → floating "💬 Comment" button appears
2. Clicking opens a side panel with the comment composer
3. Comment posted → sent to local API `POST /api/comments/<scenario>/<doc>`
4. Server appends to JSONL, returns new comment ID
5. UI highlights the anchored span in the document margin + shows in sidebar
6. Replies, resolve, delete all work on the same thread model

### Identity

- On first comment the user is prompted for a display name, stored in `localStorage`
- For shared tunnel: URL param `?reviewer=alice` prefills (convenience, not auth — password guards access)
- No OAuth, no central accounts

### Re-anchoring after regeneration

When a document is regenerated, the server walks each comment's `anchor`:
1. Try exact match of `prefix + exact + suffix` within the same `sectionId` → anchor holds
2. Try fuzzy match (Levenshtein on `exact` with tolerance)
3. If no match → comment marked `orphaned: true` and kept in a "needs reattachment" bucket

### Open questions to resolve before implementing

- [ ] Do we need real-time sync across reviewers (multiple people viewing tunnel simultaneously)? Polling is simpler; WebSocket is nicer.
- [ ] Resolve semantics: does resolve also collapse the thread, or just mark it done?
- [ ] How to surface unresolved comments to the author when they open the dashboard — badge count on scenario card?
- [ ] Should agent personas (architect/PM) be able to respond to comments as part of the review cycle?

- **Critical code fixes** — password leak, path traversal, scenario-name sanitizer, completion false-positive
- **Stable section IDs** — deterministic `id` on every HTML section (hash of title + parent path); prerequisite for commenting and re-anchoring

### 🔥 High leverage (next 2–4 weeks)

- **Text-selection comment UI** (design above) — the core collaboration feature
- **Multi-signal completion tracking** — git log / PR status / test results, not just `TODO(tag)` source scans
- **Export adapters framework** — core exports (PDF, `.docx`, single-file HTML) + adapter interface
- **First 2 adapters**: GitHub Gist (easiest) + Google Docs (widest reach)

### 📈 Medium leverage

- **Structured HTML internals** — separate data layer (`<script type="application/json">`) from presentation. Enables cleaner diffs, better regen, static chart snapshots for exports
- **Cross-scenario search + fork** — `plan_search`, `plan_fork-from-template`; learn from past plans
- **Context freshness check** — compare current git commit against context's snapshot commit; prompt re-analyze if stale
- **Scenario lifecycle tools** — `plan_rename`, `plan_archive`, `plan_delete`, `plan_next` (decision tree)
- **Reviewer extensibility** — project-local reviewer personas under `plans/.reviewers/`
- **Approval gating** — manifest records approvals per role; `plan_implementation` refuses to run without design sign-off

### 🌱 Polish / nice-to-haves

- Reduce skill count / clearer selector (11 → 4–5 entry points)
- Decision log auto-assembled from resolved comment threads
- Chart snapshots (PNG/SVG) emitted alongside HTML for export targets that can't render JS
- Session TTL + cleanup for auth sessions
- manifest.json schema + version for migration
- `plan_get_context` caching (mtime-based)
- "Export markdown summary" button on dashboard (static snapshot for pasting into Slack/PR-description)

## Technical Decisions Log

- **2026-04-17** — HTML remains source of truth (not markdown). Reason: charts/images/interactivity don't survive markdown. Collaboration is solved via built-in commenting + external adapter exports, not PR-review-of-plans.
- **2026-04-17** — Ship `dist/` to eliminate runtime `npm install`. Reason: corp network / offline failure modes; faster first-run.
- **2026-04-17** — Built-in comments use W3C TextQuoteSelector-style anchoring for regen resilience.
- **2026-04-17** — Adapters are opt-in and platform-agnostic. Word/ADO are adapters among many, not primary path.
- **2026-04-17** — Rejected `better-auth` as auth dep: requires a DB, optimized for full web-app identity, ~90% of its features (OAuth, 2FA, magic links, multi-tenant) don't apply to an in-memory per-share temporary password. Built `auth.js` (~160 LoC) in-house with HMAC-signed sessions, constant-time compare, rate limiting, rolling TTL, and loopback bypass.
- **2026-04-17** — `plan_share` never includes the password in the URL. In protected mode the plugin generates a random 16-char password and prints it alongside a clean URL; reviewer enters password + display name on a login page. Sets up comment attribution.
